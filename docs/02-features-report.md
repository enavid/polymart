# مرحله ۲ — گزارش ویژگی‌های پلتفرم فروشگاهی

> این گزارش بر پایه‌ی منابع گردآوری‌شده در [`01-research-links.md`](./01-research-links.md) نوشته شده است.
> هدف: تعیین این‌که پلتفرم فروشگاهی چندمنظوره‌ی (White-Label) ما باید چه ویژگی‌هایی داشته باشد.
>
> تاریخ: ۲۰۲۶-۰۶-۲۷

## چشم‌انداز محصول

یک **زیرساخت فروشگاهی واحد و قدرتمند** که با تغییر صرفِ تم/UI-UX بتواند هر دسته‌ی کالایی را بفروشد (قهوه، لوازم آرایشی، لوازم خودرو و …). این یعنی سه اصل بنیادین:

1. **چندمنظوره‌بودن (Multi-niche / White-label):** ظاهر، برند، زبان، ارز و حتی مجموعه‌ی ویژگی‌های هر محصول از طریق پیکربندی عوض می‌شود، نه با بازنویسی کد.
2. **مدل داده‌ی منعطف:** محصول باید بتواند هر نوع ویژگی (attribute) دلخواه را بپذیرد تا قهوه (مبدأ، درجه‌ی رست) و لوازم خودرو (سازگاری با مدل خودرو، شماره فنی) با یک مدل واحد بیان شوند.
3. **چندکاناله‌بودن (Channel-first):** از روز اول، مفهوم «کانال/فروشگاه» در دل داده باشد تا یک نصب بتواند چند ویترین مجزا را اداره کند.

---

## بخش الف — مدل داده‌ی منعطف (پایه‌ی White-Label)

این مهم‌ترین تصمیم معماری دامنه است و آن را از Saleor/Vendure/Magento می‌گیریم:

- **انواع محصول (Product Types):** ساده، متغیر (variable/configurable)، گروهی، باندل/کیت، مجازی، دانلودی/دیجیتال، کارت هدیه.
- **ویژگی‌های پویا (Attributes):** تعریف ویژگی‌های دلخواه به‌ازای هر «نوع محصول» (مثل EAV/metafield). این چیزی است که چندمنظوره‌بودن را ممکن می‌کند.
- **Variant و گزینه‌ها:** تولید SKU از ترکیب چند محور (رنگ/سایز/…) و نیز «modifier»هایی که SKU جدید نمی‌سازند (مثل حکاکی متن، آپلود فایل).
- **متادیتای JSONB:** فیلد `metadata` و `private_metadata` روی همه‌ی موجودیت‌های اصلی برای توسعه بدون تغییر اسکیما.
- **دسته‌بندی و کالکشن:** سلسله‌مراتبی + کالکشن‌های دستی و قانون‌محور (smart collection).
- **رسانه:** تصویر، ویدئو، مدل ۳بعدی/AR، متن جایگزین (alt).
- **تاکسونومی استاندارد محصول** که برای مالیات و نگاشت کانال هم استفاده می‌شود.
- **گارانتی** به‌عنوان ویژگی قابل‌فیلتر (نیاز بومی ایران).

## بخش ب — کاتالوگ و قیمت‌گذاری و پروموشن

### قیمت‌گذاری
- قیمت پایه، قیمت ویژه/حراج زمان‌دار، قیمت پلکانی/حجمی (tiered/volume).
- قیمت‌گذاری مخصوص گروه مشتری (customer group) و لیست قیمت (price list) — مهم برای B2B.
- قیمت per-channel و per-currency (هر variant چند قیمت دارد، نه یک ستون قیمت).

### موتور تخفیف و پروموشن
- تفکیک **قانون قیمت کاتالوگ** (پیش از سبد، بدون کد) از **قانون قیمت سبد** (داخل سبد، اختیاری با کد).
- انواع: درصدی، مبلغ ثابت، BOGO / Buy-X-Get-Y، ارسال رایگان، سطح سفارش/محصول/دسته، حداقل خرید، فلش‌سیل.
- کوپن: یک‌بارمصرف/چندبارمصرف، تولید دسته‌ای، محدودیت تعداد و انقضا، قوانین stacking و استثنا.
- کارت هدیه، اعتبار فروشگاهی (store credit)، امتیاز وفاداری به‌عنوان واحد ارزش.

## بخش پ — سبد خرید و چک‌اوت

- چک‌اوت تک‌صفحه‌ای + چندمرحله‌ای، سبد ماندگار بین دستگاه‌ها، mini-cart.
- خرید مهمان (guest checkout)، ساخت حساب در چک‌اوت، سفارش دستی/پیش‌فاکتور (draft order).
- چک‌اوت قابل‌توسعه: افزودن فیلد/اعتبارسنجی، سفارشی‌سازی روش‌های پرداخت/ارسال بر اساس قانون.
- گزینه‌های هدیه (بسته‌بندی + پیام)، یادداشت سفارش، ورود کوپن.
- انتخاب پرداخت اقساطی/اعتباری در همان چک‌اوت + استفاده از موجودی کیف‌پول (نیاز بومی ایران).

---

## بخش ت — پرداخت

- معماری **آداپتور درگاه (gateway adapter)** قابل‌اتصال: هیچ درگاهی hard-code نشود.
- پشتیبانی از کیف‌پول داخلی با بازپرداخت آنی به کیف‌پول.
- 3D Secure، توکنیزه‌کردن کارت، capture دستی/خودکار، پرداخت چندارزی، صورت‌حساب دوره‌ای.
- مدیریت تقلب/چارج‌بک، اسکورینگ ریسک.
- **حیاتی برای ایران:** اتصال به سوییچ شتاب/شاپرک از طریق درگاه‌های بانکی و PSPها (زرین‌پال، زیبال، آیدی‌پی، پی‌پینگ، …)؛ بدون Stripe/PayPal. آداپتور چنددرگاهی الزامی است.
- BNPL/اقساط به‌صورت first-class (دیجی‌پی، اسنپ‌پی، ازکی‌وام، تارا).
- روش‌های آفلاین: کارت‌به‌کارت/انتقال بانکی، پرداخت در محل (COD).

---

## بخش ث — حمل‌ونقل و انجام سفارش (Fulfillment)

- انواع نرخ: ثابت، رایگان، وزن‌محور، قیمت‌محور، جدول نرخ (table rate)، نرخ بلادرنگ کریر.
- مناطق ارسال (shipping zone/profile)، کلاس‌های ارسال، هزینه‌ی هندلینگ، آستانه‌ی ارسال رایگان.
- چاپ لیبل و رهگیری مرسوله، اتصال به پست/کریرهای داخلی.
- تحویل محلی (شعاعی/کدپستی)، دریافت حضوری (BOPIS)، ارسال از فروشگاه.
- انجام چندانباره و مسیریابی سفارش (نزدیک‌ترین منبع/کمینه‌کردن تقسیم مرسوله).
- درپ‌شیپینگ و 3PL.
- بازه‌ی زمانی تحویل (time-slot)، ارسال سریع/همان‌روز (الگوی دیجی‌کالا جت).

---

## بخش ج — مالیات و بین‌المللی‌سازی

- موتور مالیات قابل‌اتصال: کلاس مالیات (محصول/مشتری/ارسال)، منطقه و نرخ بر اساس کشور/استان/کدپستی.
- نمایش قیمت با/بدون مالیات بر اساس منطقه.
- **چندارزی:** ارز نمایش در برابر ارز تراکنش، نرخ خودکار/دستی، گردکردن قیمت per-market.
- **چندزبانه:** ترجمه‌ی کاتالوگ/چک‌اوت/ایمیل/UI، محتوای per-locale.
- SEO بین‌المللی: hreflang، استراتژی subfolder/subdomain.
- **بومی ایران:** UI راست‌به‌چپ فارسی، تقویم جلالی/شمسی برای همه‌ی تاریخ‌ها و گزارش‌ها، نمایش تومان در برابر حسابداری ریال، هویت مبتنی بر شماره موبایل.

## بخش چ — موجودی و انبار

- ردیابی موجودی per-variant، وضعیت‌های on_hand/available/committed/reserved/incoming/damaged/safety-stock.
- موجودی چندمنبعی (MSI): منبع‌ها (انبار/فروشگاه) → stock نگاشت‌شده به کانال فروش.
- رزرو موجودی هنگام چک‌اوت (جلوگیری از فروش بیش از موجودی بدون قفل).
- الگوریتم انتخاب منبع (priority/distance)، انتقال بین انبارها، دریافت با بارکد.
- آستانه و هشدار کم‌موجود/ناموجود، پیش‌خرید (backorder)، نگه‌داشت موجودی برای سفارش پرداخت‌نشده.
- روش‌های ارزش‌گذاری (FIFO/Average/…)، شمارش دوره‌ای، استراتژی برداشت (FIFO/FEFO).

## بخش ح — بازاریابی و مرچندایزینگ

- **SEO:** عنوان/متا/slug قابل‌ویرایش، ریدایرکت ۳۰۱، canonical، sitemap/robots خودکار، داده‌ی ساختاریافته JSON-LD.
- **بازیابی سبد رهاشده:** ایمیل/پیامک تک و چندمرحله‌ای، درج خودکار کوپن، گزارش درآمد بازیافت‌شده.
- **ایمیل/پیامک مارکتینگ:** فلوهای خودکار (خوش‌آمد، win-back، back-in-stock).
- **وفاداری/امتیاز:** امتیاز بابت خرید/ثبت‌نام/نظر/معرفی، سطوح (tier)، گیمیفیکیشن (الگوی DigiClub).
- **نظرات و امتیازدهی:** نظر + پرسش‌وپاسخ، پاسخ فروشنده، UGC.
- **توصیه‌گر محصول:** قانون‌محور + هوش مصنوعی («با هم خریده‌شده»، «پرطرفدار»، «برای شما»).
- **جست‌وجوی هوشمند و فاست:** typo-tolerance، مترادف، autocomplete.
- **CMS/محتوا:** page builder، بلاگ، بنر منیجر، بلوک‌های هدف‌مند.
- **چت/پشتیبانی زنده**، کامرس اجتماعی (الگوی چت فروشنده باسلام).

## بخش خ — تحلیل و گزارش

- داشبورد بلادرنگ، فروش/سفارش/درآمد، AOV، نرخ تبدیل، عملکرد محصول/دسته.
- گزارش‌ها: اکتساب/رفتار، مشتری و cohort، موجودی (ABC، sell-through)، کوپن، مالیات، سبد رهاشده.
- گزارش‌ساز سفارشی، گزارش سود/مارجین (COGS)، اکسپورت زمان‌بندی‌شده، مقایسه‌ی دوره‌ای و YoY.
- لاگ ممیزی (audit log): چه کسی، چه زمانی، چه چیزی را تغییر داد.

## بخش د — چندفروشگاهی / چندمستأجری / کانال‌ها

- چند ویترین از یک کاتالوگ/حساب واحد؛ هر کانال: دامنه/تم/ارز/کاتالوگ/قیمت جدا.
- مدیریت مرکزی سازمان (کاربر/صورت‌حساب/امنیت بین فروشگاه‌ها).
- کانال‌ها: فروشگاه آنلاین، headless، POS، Buy Button، مارکت‌پلیس‌ها، شبکه‌ی اجتماعی، Google Shopping، فید.
- **مدل مارکت‌پلیس چندفروشنده (اختیاری/فاز بعد):** ویترین per-seller، آنبوردینگ، کارمزد، داشبورد فروشنده، اعتبار/امتیاز فروشنده.
- اتوماسیون گردش‌کار (الگوی Shopify Flow): trigger/condition/action بدون کد.

## بخش ذ — B2B (فاز پیشرفته)

- حساب شرکتی با سلسله‌مراتب، چند خریدار، چند آدرس ارسال.
- نقش‌ها و گردش‌کار تأیید (approval workflow).
- پورتال خریدار: سفارش‌ها، پیش‌فاکتور، فاکتور، لیست‌ها.
- کاتالوگ/قیمت اختصاصی per-account، پیش‌فاکتور قابل‌مذاکره (RFQ).
- سفارش سریع با SKU/CSV، لیست خرید مجدد.
- اعتبار شرکتی/خط اعتباری/پرداخت اقساطی (Net terms).

## بخش ر — اشتراک و پرداخت دوره‌ای (فاز پیشرفته)

- selling plan (subscribe-and-save، prepaid)، قرارداد اشتراک، سیاست صورت‌حساب/تحویل، dunning.
- مدل‌ها: دوره‌ای (هفتگی/ماهانه/سالانه)، مصرف‌محور، با هزینه‌ی ثبت‌نام/دوره‌ی آزمایشی.
- پورتال سلف‌سرویس مشتری: تغییر SKU/تعداد/دوره/آدرس، pause/skip، ارتقا/تنزل، لغو.

## بخش ز — پنل ادمین / CRM

> این بخش پاسخ مستقیم به خواستهٔ «پنل ادمین/CRM قدرتمند» است.

### مدیریت سفارش و انجام آن
- لیست و نمای فیلترشونده‌ی سفارش بر اساس وضعیت سفارش/پرداخت/انجام/مرجوعی.
- جزئیات سفارش: اقلام، مشتری، پرداخت، timeline/یادداشت، تگ داخلی.
- ویرایش سفارش، عملیات پرداخت (authorize/capture/void/mark-paid)، انجام جزئی/کامل، چاپ لیبل، مارک تحویل، لغو + بازگردانی موجودی.

### مرجوعی/RMA و بازپرداخت
- ایجاد مرجوعی (سلف‌سرویس مشتری یا توسط کارمند)، قوانین مرجوعی (پنجره/هزینه).
- فلو RMA: درخواست → تأیید → لیبل برگشت → دریافت → بازرسی → اصلاح موجودی → بازپرداخت.
- بازپرداخت کامل/جزئی به روش اولیه یا اعتبار فروشگاهی.

### CRM و مدیریت مشتری
- پروفایل مشتری: تماس، آدرس‌ها، تاریخچه‌ی سفارش، ارزش طول عمر (LTV)، موجودی کیف‌پول.
- **بخش‌بندی RFM** (Champions، Loyal، At-Risk، Lost، …).
- تیکتینگ/پشتیبانی omnichannel با context کامل مشتری.
- ابزار GDPR/حریم خصوصی: درخواست/پاک‌کردن داده، ادغام تکراری‌ها.

### سایر بخش‌های پنل
- مدیریت محصول/کاتالوگ (با تفکیک دسترسی دیدن «قیمت تمام‌شده»).
- مدیریت انبار/موجودی (WMS).
- مدیریت پروموشن/کوپن، CMS، داشبورد گزارش.
- لاگ فعالیت کارکنان (audit log) برای همه‌ی تغییرات حساس.

## بخش ژ — کاربران و سطوح دسترسی (RBAC)

> پاسخ مستقیم به خواستهٔ «کاربرها با سطح دسترسی‌های مختلف».

### مدل دسترسی (دو لایه)
1. **لایه‌ی نقش‌محور:** `Group` جنگو به‌عنوان نقش + Permissionهای مدل‌سطح (شامل permissionهای سفارشی مثل `can_refund_order`, `can_capture_payment`, `can_fulfill_order`, `can_view_cost`). دسترسی‌ها بین گروه‌ها جمع می‌شوند.
2. **لایه‌ی سطح‌شیء/scope:** با `django-guardian` برای محدودکردن کارمند به زیرمجموعه (مثلاً کارمند انبار فقط انبار خودش). مشابه scope کانال در Saleor.

اعمال در API با DRF (`DjangoModelPermissions` + `DjangoObjectPermissions` + فیلتر guardian)، **secure-by-default** (هر اندپوینت بدون permission صریح، بسته است — الگوی Vendure).

### دامنه‌های دسترسی (نمونه codename)
`MANAGE_ORDERS, FULFILL_ORDERS, REFUND_ORDERS, CAPTURE_PAYMENTS, MANAGE_RETURNS, MANAGE_PRODUCTS, VIEW_PRODUCT_COST, MANAGE_INVENTORY, MANAGE_PROMOTIONS, MANAGE_CUSTOMERS, ERASE_CUSTOMER_PII, MANAGE_SUPPORT_TICKETS, MANAGE_CMS, VIEW_REPORTS, MANAGE_STAFF, VIEW_AUDIT_LOG, MANAGE_SETTINGS`

### نقش‌های پیشنهادی کارکنان

| نقش | خلاصه‌ی دسترسی | محدوده |
|---|---|---|
| Super Admin / Owner | همه‌چیز (شامل MANAGE_STAFF، MANAGE_SETTINGS، VIEW_AUDIT_LOG) | کل سیستم |
| Store Manager | همه به‌جز مدیریت کارکنان/تنظیمات/مالی؛ گزارش‌ها | کل/per-channel |
| Order Processor | مدیریت و ویرایش سفارش، capture پرداخت، ایجاد انجام سفارش | خط سفارش |
| Warehouse / Fulfillment | pick/pack/ship، موجودی، دریافت مرجوعی | محدود به انبار خود |
| Support Agent | مشتری، تیکت، شروع مرجوعی، بازپرداخت محدود (به اعتبار) | محدود به تیکت/مشتری |
| Marketing | پروموشن، CMS، سگمنت، گزارش مارکتینگ | کل/per-channel |
| Merchandiser / Catalog | محصول، قیمت تمام‌شده، CMS | کل/per-channel |
| Finance | بازپرداخت کامل، capture، گزارش مالی، disputes | کل |
| Read-only / Analyst | فقط مشاهده + گزارش | کل |

> دسترسی‌های حساس عمداً جدا شده‌اند: `VIEW_PRODUCT_COST`، `REFUND_ORDERS` (کامل)، `ERASE_CUSTOMER_PII`، `MANAGE_STAFF`، `MANAGE_SETTINGS`، `VIEW_AUDIT_LOG`.

## بخش س — معماری و توسعه‌پذیری

- **Channel به‌عنوان موجودیت درجه‌یک** از روز اول (ارز/زبان/قیمت/مالیات/انبار per-channel).
- جداسازی فیزیکی transport از دامنه (الگوی Saleor): دامنه هیچ‌وقت DRF/Graphene را import نمی‌کند.
- لایه‌ی use-case/interactor صریح (الگوی Medusa Workflows + Application layer سیلیوس).
- آداپتور/پورت برای زیرساخت (پرداخت، کش، event bus، جست‌وجو، فایل، اعلان) با وارونگی وابستگی.
- توسعه‌پذیری: ترجیح وب‌هوک sync/async (الگوی Saleor) + الگوی Strategy و رجیستری برای نقاط تغییر داخلی.
- event bus + کارهای پس‌زمینه (Celery) برای side-effectها.

## بخش ش — تفکیک «هسته» در برابر «تم»

برای تحقق White-Label واقعی:

- **هسته (Core / Backend):** کاملاً headless و UI-less. فقط API + دامنه. تغییر niche آن را عوض نمی‌کند.
- **تم/ویترین (Frontend):** بازاسکین‌پذیر با **معماری توکن سه‌لایه** (primitive → semantic → brand/niche theme).
  - تحویل تم: build-time per-tenant یا runtime (تزریق CSS variable بر اساس دامنه/زیردامنه).
  - لایه‌ی API تایپ‌دار که UI هرگز مستقیماً شکل DRF را لمس نکند (provider-swap الگوی vercel/commerce).
  - i18n + RTL با logical properties؛ فارسی first-class (فونت Vazirmatn، اعداد فارسی، تقویم جلالی).

## شکاف‌ها و فرصت‌های تمایز

این موارد در پلتفرم‌های غربی ضعیف/گران‌اند و فرصت تمایز ما هستند (به‌ویژه برای بازار ایران):

- اشتراک نیتیو، وفاداری و نظرات نیتیو (نه افزونه‌ی گران).
- آبستره‌سازی چنددرگاهی شاپرک/PSP (بدون Stripe/PayPal).
- پرداخت اقساطی/BNPL به‌صورت first-class در چک‌اوت.
- کیف‌پول داخلی با بازپرداخت آنی.
- احراز هویت با OTP موبایل + پیامک مارکتینگ.
- تقویم جلالی، UI راست‌به‌چپ فارسی، نمایش تومان/حسابداری ریال.
- ردیابی گارانتی، کامرس اجتماعی، اتصال به مقایسه‌گرهای قیمت (ترب/ایمالز).
- مدل مارکت‌پلیس چندفروشنده + اپ فروشنده (فاز بعد).

## جمع‌بندی

پلتفرم باید **هسته‌ی headless مبتنی بر Django + Clean Architecture** با مدل داده‌ی فوق‌منعطف (attribute پویا، metadata، channel-first) باشد، و یک **ویترین React بازاسکین‌پذیر** با معماری توکن سه‌لایه روی آن. این دو لایه با یک API تایپ‌دار از هم جدا می‌شوند تا تغییر niche فقط یعنی «تم + پیکربندی جدید»، نه کد جدید. RBAC دولایه (نقش + scope) خط لوله‌ی پردازش سفارش را میان کارکنان توزیع می‌کند. فازبندی پیاده‌سازی در [`03-roadmap.md`](./03-roadmap.md) آمده است.
